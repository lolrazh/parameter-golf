## Bugs & Issues Encountered

1. **N-gram p_val > 1 from hash collisions** — `full_counts[hash]` can exceed `ctx_counts[hash]` when different keys map to same bucket.
   - **Fix:** `.clamp(max=1.0)` on p_val.

2. **NGRAM_MAX_ORDER > 7 crashes** — Only 7 primes in the hash function.
   - **Fix:** Extended primes list to 9, added assert.

3. **DDP cache fragmentation** — Each rank has 1/8th of scored tokens in its cache. Attempted sync via all_reduce but discovered double-counting bug (synced values get re-summed).
   - **Workaround:** Accepted as limitation. Each rank still sees ~7.7M tokens, sufficient for n-gram statistics.

4. **Redundant sliding window eval added ~75-98s to eval time** — Standalone sliding window ran before TTT, pushing logged eval to 651-679s (over 600s budget).
   - **Resolution:** Submitted original code with honest transparency note in README. Without the redundant eval, times are 576-581s (within budget). The redundant eval doesn't affect BPB — TTT's built-in scoring is the reported number.

5. **Seed 1337 slower than 42/2025** — 104ms/step vs ~85-93ms for the other seeds. Unknown cause — possibly GPU thermal throttling or NCCL variability. Resulted in fewer steps (5,735 vs 6,446-6,799).
   - **Impact:** Seed 1337 had the highest BPB (0.9977) of the three, consistent with fewer training steps.

## Key Learnings
- **N-gram cache is the biggest single lever.** ~0.15 BPB gain over post-quant, dwarfing everything else (QAT, mixed quant, activation changes). The FineWeb val set has massive n-gram repetition.
- **TTT + n-gram compound.** Early chunks get weak n-gram stats and an unadapted model. Late chunks get rich n-gram context AND a model adapted by hundreds of SGD steps. BPB drops from 1.21 to 0.99 across chunks.
- **DDP cache fragmentation is a real limitation.** Single-GPU would have stronger n-gram cache (all 62M tokens in one table). Worth solving with proper delta-based sync for future submissions.
- **Timing budgets need upfront planning, not afterthoughts.** The redundant sliding window eval was a careless oversight. Should have profiled eval phases before the prod run.
- **Submit what you actually ran.** Tempting to patch the code post-hoc and pretend the timing was always clean. Transparency note in README is the right call.

## Architecture Decisions
- **Fixed alpha=0.2 (not entropy-adaptive)** — Conservative, clearly legal, still gives massive gains. Adaptive alpha would squeeze more BPB but risks legality questions.
- **Original code in submission (not patched)** — Honest. The redundant eval is a cosmetic issue, not a correctness issue. Reviewers can verify the fix is trivial.
- **N-gram on CPU, blend on GPU** — Hash tables are 192MB, too large for GPU memory alongside the model. CPU lookups are fast enough (~microseconds per position). Only small tensors transfer to GPU for blending.
