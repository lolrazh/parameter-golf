## Bugs & Issues Encountered
1. **Artifact over 16MB on first run** — Uniform int6 quantization produced 16,718,251 bytes with sp4096 vocab.
   - **Fix:** Added int5 MLP quantization (`bits = 5 if ".mlp." in name else 6`). Reduced to ~14.7MB.
2. **sp4096 data not in official manifest** — `cached_challenge_fineweb.py --variant sp4096` fails.
   - **Fix:** Custom download script pulling from `sproos/parameter-golf-tokenizers` HuggingFace repo.
3. **SSH pipe buffering** — `python3 ... | tail -25` never showed output because Python stdout was buffered and `tail` waits for all output.
   - **Fix:** Use `PYTHONUNBUFFERED=1` + write to files on GPU, then cat results. Always use `nohup` for long runs.
4. **SSH connection drops** — Thunder Compute 1xH100 had transient network failures mid-run.
   - **Fix:** Always use `nohup ... > logfile 2>&1 &` so runs survive SSH drops.
5. **First run got only 6,130 steps vs second pod's 8,240** — Same config, different step counts.
   - **Root cause:** First pod was RunPod with higher latency per step (97.8ms avg including slow compile). Second pod had faster interconnect (72.8ms stable). Hardware variance matters.
6. **`sota_train_gpt.py` was deleted** — Source of fused QKV + batched Muon was removed in cleanup.
   - **Fix:** Recovered from git history via `git show f8a1863^:sota_train_gpt.py`.

## Key Learnings
- **Int5 MLP is not free** — 0.040 BPB quant gap vs 0.013 for uniform int6. But TTT (-0.235 BPB) makes it irrelevant.
- **FA3 is faster than SDPA on 1xH100 now** — Previous testing showed FA3 slower (182 vs 132ms). With fused QKV producing natural [B,T,H,D] layout, FA3 skips the transpose and wins (153ms).
- **Module-level env var toggles get constant-folded by torch.compile** — Dead branches eliminated at compile time, zero runtime overhead.
- **72ms/step on 8xH100 vs SOTA's 67ms** — Gap is mostly from sp4096 vocab (4x larger embedding table). Worth it for 35% more bytes/token.
- **3 seeds are remarkably consistent** — std 0.0016 BPB across seeds. The training pipeline is stable.
- **zstd-22 is already optimal** — Best general-purpose compressor for quantized weights. All SOTA entries use it. No better alternative.

## Architecture Decisions
- **Int5 MLP over uniform int6** — Trades 0.027 BPB in quant quality for 2MB artifact savings. Enables fitting under 16MB with sp4096 at 10 layers.
- **warmdown=2000 not 3000** — At 8,240 steps, 2000 = 24% at low LR (optimal). 3000 would be 36% (too aggressive, hurt on prior s3 run).
- **3 standard seeds (42, 1337, 2024)** — Matches SOTA submission convention. Enables direct comparison.
- **nohup + file logging** — SSH connections to GPU pods are unreliable. All runs must survive disconnects.

## CRITICAL: 3-Epoch TTT Is INVALID

**The 0.9619 BPB results above used 3-epoch TTT which violates competition rules.** Will Depue (organizer) clarified in issue #402: "you are only allowed to test-time train on validation set tokens you've already evaluated your model on." Multi-epoch TTT trains on tokens before they're scored on epochs 0 and 1. PROTEUS got taken down for the same reason.

**Only 1-epoch score-then-train is legal:** score chunk → train on it → next chunk. Each token graded before being used for training. This matches merged PR #77 (LoRA TTT, 1.1928 BPB).

**Impact:** 1-epoch TTT gain is estimated at -0.03 to -0.05 BPB (vs -0.235 for 3-epoch). Our post-quant BPB (~1.20) + 1-epoch TTT → estimated ~1.15-1.17. Current SOTA is 1.1228 (signalrush). We likely can't beat SOTA with 1-epoch TTT alone — need additional improvements.

**New improvements implemented (pending testing):**
- GPTQ-lite: per-row optimal clip percentile search (5 candidates, min MSE). Zero training cost.
- QAT: STE fake int6/int5 quantization when lr_scale < QAT_START_FRAC. Shrinks quant gap.
- Sliding window eval (EVAL_STRIDE=64): more context per token during eval.
- Seq length 2048: longer training context.
